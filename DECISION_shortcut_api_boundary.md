# 決定記録: .rjs配送経路とShortcutApi拡張の境界線

## 結論

- **`.rjs`スクリプトの「テキストをどこから取得するか」**(assets同梱、将来的なトンネル経由等)は自由に差し替えてよい。`RjsShortcutScanner`が返す`Map<String, String>`を`DeviceScriptEngine.registerAll()`に渡す、という形さえ守れば経路は問わない
- **`ShortcutApi`が公開するJava操作の集合そのもの**は、実行時に外部から追加・変更できるようにしてはならない。新しいJava操作が必要になった場合は、必ず`ShortcutApi.kt`にメソッドを追加し、その都度安全性を吟味した上でKotlin側を再コンパイル・再配布する

## 背景・却下した案

「APIを追加するための一方通行トンネル(RJS Generator専用)」という案を検討したが却下した。理由:

1. トンネル経由でJava操作そのものを注入できるようにすると、これまでの安全設計(`SAFETY_AND_SCOPE_BOUNDARIES.md`、`DECISION_device_engine_rhino.md`)の前提である「危険なことができるAPIをそもそも公開しない」という静的な絞り込みが、実行時に動的へ変わってしまう
2. トンネルの通信経路や送信元が想定外の相手に渡った場合、その瞬間に任意のJava操作が実行可能になる。JAScriptの`ctx`フルアクセス方式が「最初から全スクリプトが無制限アクセスを持つ静的な危険」であるのに対し、この案は「いつ・誰が権限範囲を書き換えるかわからない動的な危険」が上乗せされ、むしろ検知・監査の観点でJAScript方式より厄介になりうる
3. ローカルサーバー/トンネルは同一端末上の他アプリやcontent側(ページ由来の信頼できないJS)からも到達しうる経路になり得るため、「非公開だから安全」という前提は脅威モデル上成立しない

## 今後、新しいJava操作を追加したくなった場合の手順(再確認)

1. `ShortcutApi.kt`にメソッドを1つ追加する
2. そのメソッドが安全か吟味する(限定APIの原則を満たしているか)
3. アプリを再ビルド・再配布する
4. `.rjs`側から`shortcuts.新メソッド名(...)`として呼ぶ

この摩擦(再コンパイル前提)は意図的なものであり、解消しようとする方向の設計変更(トンネル等)は行わない。
